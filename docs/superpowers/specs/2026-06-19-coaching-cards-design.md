# 코칭 카드 강화 — 한글→영어 표현 + 영어 다듬기

작성일: 2026-06-19

## 배경 / 문제

매일 아침 사용자는 `daily/<날짜>/digest.md` 만 읽는다(CLAUDE.md 규칙).
그런데 학습 가치가 큰 두 가지가 digest 에서 보이지 않는다:

1. **한글→영어 코칭** — 프롬프트 7번에 지시는 있으나 *출력 위치가 지정돼 있지 않아*
   LLM 이 임의로 `new-expressions.md` 맨 아래에 붙였다(위치 불안정 + digest 부재).
2. **영어 다듬기(문법 정정 + 더 나은 표현)** — 프롬프트에 단계 자체가 없다(완전 누락).

입력 배치에는 이미 `[user]` / `[assistant]` role 태그가 붙어 있어
(`collect.py` 의 `f"[{rec['role']}] {rec['text']}"`) 사용자가 직접 쓴 글을 구분할 수 있다.

## 목표

- 사용자가 한국어로 쓴 내용 중 학습 가치 있는 것을 "자연스러운 영어"로 표현해 보여준다.
- 사용자가 직접 쓴 영어를 다듬는다: 오류가 있으면 정정하고, **오류가 없어도 항상 한 단계 위의 더 나은 표현을 제안**한다.
- 두 카드를 매일 보는 `digest.md` 에서 가시화한다.

## 비목표 (YAGNI)

- 파이썬 파이프라인 코드 변경 없음. `finalize` 의 `git add -A` 가 새 파일을 자동 커밋한다.
- collect/transcripts 동작 변경 없음(이미 role 태그·소스 수집 충분).
- 별도 설정 플래그/토글 없음.

## 소스 판정 기준

| 카드 | 소스 |
|------|------|
| 한글→영어 | transcript 의 `[user]` 한국어 + `spool:` 노트 |
| 영어 다듬기 | transcript 의 `[user]` 영어 + `spool:` 영어 |

`[assistant]` 메시지와 `repo:` 문서는 코칭 소스에서 제외(내가 쓴 글이 아님).
`repo:`/`[assistant]` 영어는 기존대로 "표현 추출"·"정독 단락" 에서만 다룬다.

## 설계

### A. 새 출력 파일 `daily/<날짜>/coaching.md`

두 카드 타입을 한 파일에 모은다. 한글→영어 카드는 `new-expressions.md` 에서 이리로 이동.

```markdown
# <날짜> — 코칭

## 한글→영어
### 카드 1 — <짧은 제목>
- 내가 쓴 한글: "..."   (출처: transcript:... 또는 spool:...)
- 자연스러운 영어: ...
- 왜 이렇게: ...

## 영어 다듬기
### 카드 1 — <짧은 제목>
- 내가 쓴 영어: "..."   (출처: transcript:[user] ... 또는 spool:...)
- 정정: ...            (문법 오류가 있을 때만; 틀린 부분 + 규칙 한 줄)
- 더 나은 표현: ...     (항상; 자연스러움/격식 한 단계 위 버전)
- 왜: ...
```

규칙:
- 한쪽 카드가 하나도 없으면 그 `##` 섹션은 통째로 생략.
- 두 종류 모두 없으면 `coaching.md` 자체를 만들지 않고, digest 코칭 섹션도 생략.
- `정정` 줄은 실제 오류가 있을 때만 출력(없으면 그 줄 생략). `더 나은 표현` 은 항상 출력.
- 추출 충실 원칙 유지: 없는 오류를 지어내지 않는다. "내가 쓴 한글/영어" 는 배치에 실제 등장한 사용자 텍스트만 인용.

### B. `prompts/process.md` 수정

작업(`## 작업`) 섹션:
1. 7번(한글→영어 코칭): 소스를 명시(`[user]` 한국어 + spool) 하고 출력 위치를 `coaching.md` 로 지정.
2. **새 8번(영어 다듬기)** 추가: `[user]` 영어 + spool 영어를 다듬는다.
   오류가 있으면 정정, 없어도 더 나은 표현을 항상 제안. 사용자 영어가 배치에 없으면 생략.
3. 기존 8·9번(중복 제거·spool 답변)을 9·10번으로 재번호.

출력(`## 출력`) 섹션:
4. `coaching.md` 항목 신설(위 A 형식).
5. `new-expressions.md` 설명에서 한글→영어 코칭 카드 제거(이제 coaching.md 로 감).
6. `digest.md` 에 `### 오늘의 코칭` 추가 — 코칭 카드가 있을 때 한글→영어/영어 다듬기에서
   각 1~2개를 1줄씩 요약하고 `coaching.md` 를 가리킨다. 없으면 섹션 생략.

원칙(`## 원칙`) 섹션:
7. 코칭 소스 제한(`[user]`/spool 만)과 "오류는 지어내지 않되 더 나은 표현은 항상" 규칙을 한 줄 추가.

### C. 코드 변경

없음.

## 데이터 흐름

```
collect → batch-<날짜>.md  ([user]/[assistant]/repo: 태그 포함)
   → claude -p (process.md)
        ├─ new-expressions.md   (표현; 한글→영어 카드 제거됨)
        ├─ reading.md           (정독 단락; 변화 없음)
        ├─ coaching.md          (신규: 한글→영어 + 영어 다듬기)
        └─ digest.md            (### 오늘의 코칭 섹션 추가)
   → finalize: git add -A → commit/push  (coaching.md 자동 포함)
```

## 검증

자동화된 단위 테스트로는 LLM 출력을 검증하기 어렵다. 검증은 실행 결과로 한다:

1. `prompts/process.md` 의 작업/출력/원칙 번호 정합성 육안 확인(중복 번호 없음).
2. 대화형 `/study` 또는 수동 `pipeline.run` 1회 실행 후:
   - `daily/<날짜>/coaching.md` 가 생성되고 두 섹션 형식이 맞는지.
   - 한글→영어 카드가 `new-expressions.md` 에 더 이상 없는지.
   - `digest.md` 에 `### 오늘의 코칭` 이 있고 coaching.md 를 가리키는지.
   - 사용자 영어가 배치에 없던 날엔 영어 다듬기 섹션이 깔끔히 생략되는지.
